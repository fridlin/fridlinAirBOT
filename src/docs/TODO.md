## 0. Output Flow (CANONICAL ‚Äî DO NOT BREAK)

This project uses a strictly defined multi-message output flow.
Each step is a separate message with its own UX and text layer.

The flow MUST NOT be merged, reordered, or simplified without explicit decision.

### Output sequence:

0. Any text input
   - Any text that is NOT a known command or button
   - Is treated as `/start`

1. Start / Onboarding
   - Greeting message
   - Language selection

2. Location request
   - Explicit request to send GPS location

3. Timezone calculation (UX message)
   - Short message: "Determining your timezone‚Ä¶"
   - Small artificial delay allowed (UX magic)

4. Timezone result
   - Message showing resolved timezone

5. Micro forecast calculation (UX message)
   - Message: "Calculating micro forecast for your location‚Ä¶"

6. Micro forecast output
   - Main forecast message
   - Built via microForecastUI
   - Must be modular and configurable (easy to add/remove parameters)
   - UX and text are fully separated from logic

7. Alarm message (optional)
   - Only if meteorological danger exists
   - Separate message
   - Minimal and strong UX

8. Warning + recommendations message (optional)
   - Comfort warnings
   - Clothing / attention / safety notes
   - Separate message
   - Extensible via UX and text configs

### Architecture rules:

- Each output step = separate message
- Each message has:
  - Logic layer (data only)
  - UX layer (presentation only)
  - Text layer (language-specific)
- Adding a new language MUST NOT require code changes
- UX changes MUST NOT affect logic
